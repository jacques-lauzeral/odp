UI Design Proposal
for ODIP Space

This document proposes the design of ODIP Space — the EUROCONTROL Network Management application for managing and publishing the Operational Development and Implementation Plan. It covers identity, navigation, page layout, and interaction design for all user-facing elements.

Prepared by Yves Steyt / ACD Division
v1.42 — April 2026
Proposed design — all pages and interactions

Introduction

This document grew out of hands-on work with an existing partial implementation of ODIP Space. Direct use revealed gaps, inconsistencies, and missing features that, taken together, pointed towards a significant evolution of the application's design. This document reflects that evolution.

A central conviction drives the design: visual identity and user experience are not cosmetic concerns — they are preconditions for acceptance. ODIP Space is heading towards its first production release and will be used by a broad community of operational stakeholders, including external organisations across European air traffic management. A tool that looks provisional will be perceived as provisional. First impressions shape trust, and trust shapes adoption. The interface must convey quality and care from the moment a user lands on the Home page.

This means that visual identity, consistent layout, and coherent interaction patterns are not post-release refinements. They are release requirements. The design proposals in this document — the logo, the header, the navigation model, the page structure — should be in place before ODIP Space is shared externally.

The document is structured as a complete design specification. It is intended to serve as the primary reference for implementation. Where topics remain open, they are explicitly flagged in §0.

3 — Site plan diagram

ODIP Space is a single-page application built around a persistent header and a page stack.

The header is the structural spine of the application. It is present on every page without exception and carries four distinct responsibilities. The left block identifies the application and the current user: it shows the ODIP Space logo, the user's name, organisation, and role badge. Its appearance changes depending on context — a passive state when no version is selected, a read-only state when browsing a published edition, a read/write state when working on the live dataset, and a red patching state when a patch operation is in progress. The navigation bar occupies the centre of the header and provides access to four tabs at any time, covering five distinct areas. Its tabs are selectively visible depending on the user's role and the active version context — internal users see more tabs than external users, and some tabs are only available when the dataset is selected. The Stack indicator sits in the header and tracks the user's navigation history: it shows how many pages deep the user has navigated, signals unsaved changes in amber, and offers a dropdown to jump directly to any level in the stack or clear it entirely. The contextual help trigger is also anchored in the header and opens a floating singleton help window whose content adapts to the page and field the user is currently on.

The navigation bar shows four tabs at any time, drawn from five distinct areas. Elaboration and Explore are mutually exclusive — only one is shown depending on whether a dataset or a published edition is selected; neither appears when no version is selected. The other three tabs are fixed. Home is the entry point — its sole purpose is version selection. Elaboration is the dataset authoring workspace, available to internal users only, organised into sub-tabs: Dataset description · O*s · Plan · Quality · Setup. Explore is the read-only counterpart to Elaboration, operating on published editions rather than the live dataset — it mirrors the O*s, Plan, and Quality sub-tabs without editing capability; Quality in Explore is read-only and visible to active users only. Conversations is the collaborative feedback surface, covering threads at application level, edition level, and individual O* level. Management is the administration area, restricted to integrators, version-independent — covers Editions lifecycle (baseline creation, publish/unpublish, patching) and Admin (Monitoring, Users, Log, Backup).

From any context, navigating into an O*, a wave, or a conversation thread pushes a detail page on top of the stack. Detail pages are not separate destinations in the navigation bar — they are contextual overlays reachable from result sets, plan views, or links anywhere in the application. The stack can hold multiple levels; the Back control always returns to the previous level.

5 — Application behaviour — universal rules

Foundational rules that apply universally across all pages, all roles, and all contexts. Everything else in this document derives from or is consistent with these principles.

DRAFT visibility — the authoring boundary

DRAFT is the authoring workspace. Authors elaborate O*s freely without their work-in-progress being exposed to anyone outside the dataset. An O* in DRAFT state is never visible outside the dataset — not in editions, not to external users, not in any generated document, not in any plan view, not in any text search run against an edition.

Only when an O* reaches ADVANCED or MATURE does it enter the public-facing lifecycle and become eligible for editions, prioritisation, and external consultation. This is not a display rule — it is a lifecycle boundary.

6.4 — Navigation model

Universal principle

Why a stack, not a breadcrumb

ODIP Space uses a single-page, single-tab navigation model based on a page stack. There are no new browser tabs, no independent pages, no browser Back/Forward navigation (blocked). Everything happens within one browser tab. The stack is global — shared across all top-level navigation tabs (Home, Elaboration, Explore, Conversations, Management). Management is version-independent and does not use the stack for its internal navigation — it uses a persistent left nav instead.

A breadcrumb — a horizontal trail showing the path from root to current page — only works in hierarchical navigation, where there is a single canonical path from any node to the root. SharePoint is the canonical example, and its breadcrumb works poorly there too, precisely because users navigate via links and shortcuts rather than descending the hierarchy.

ODIP Space navigation is a graph, not a tree. The same OR detail page can be reached from a result set, from an ON Plan tab, from an OC Implemented ORs table, from the evolutions panel, from a dependency list — all legitimate paths, producing the same page at the top of the stack with completely different pages below it. A breadcrumb would show one arbitrary path and imply it is the definitive one. The stack records your actual path — personal and dynamic. Two users looking at the same OR detail page may have completely different stacks underneath it, and both are valid. The stack indicator exposes your path, not the site's structure.

The page stack

Home (with the currently selected ODIP version) is always at the bottom of the stack and is never removed. Every subsequent action pushes a new page on top. Closing a page removes it from the top and reveals whatever was underneath, exactly as the user left it — same scroll position, same state.

The stack can grow several levels deep. For example: Home → result set → OC detail → OR detail → ON detail. Each ← Back or × peels one level off.

What pushes a new page onto the stack: clicking any named object link (O\*, wave, edition, …) or triggering a "New [object]" action (New ON, New wave, etc.). Every named object in the application is always a link — regardless of where it appears.

Gallery-mode walking and edit mode

The five scenarios below are illustrated with O*s, but the same rules apply wherever a set of objects can potentially be visited or edited one after the other — wave result sets, edition lists, and any future object types that have a result set.

Scenario 1 — Click a row title → gallery level pushed. Clicking a row title in a result set pushes a single stack level that contains both the result set and the detail page. The detail page is the visible surface; the result set is retained underneath. This is one level, not two. The position counter (e.g. "3 of 12") shows the current position within the result set. "← Back to result set" returns to the list view within that same level — it does not go up a stack level.

Scenario 2 — Walk with chevrons. The ‹ and › chevrons fixed to the left and right viewport edges step through the result set without pushing a new level. Each step replaces the visible detail page within the existing gallery level. Stack depth does not change.

Scenario 3 — Row-level Edit (hover action). Each result set row exposes an Edit action on hover. Clicking it opens the object directly in edit mode as a plain detail page — no gallery level, no chevrons, no "← Back to result set". This is a distinct entry point from clicking the row title: it is intended for editing a specific object, not for browsing the set. ← Back returns to the result set. Save or Cancel return the page to read mode; ← Back then returns to the result set.

Scenario 4 — Follow a link inside a detail page → new level pushed. Clicking any named object link inside a detail page (e.g. an ON title in an OR detail page) pushes a new stack level on top of the gallery level. The new page has no result set context — no chevrons, no "← Back to result set". ← Back from that page returns to the detail page below.

Scenario 5 — Object opened outside a result set. When an object is reached by clicking a link that is not a result set row (e.g. an OR link in an OC Implemented ORs table), no result set is retained. No chevrons. No "← Back to result set". ← Back returns to the page below in the stack. Edit mode works identically — same amber sub-nav, same Save/Cancel behaviour, same stack depth rule.

8.1 — Contextual help — architecture and UX

Design decisions · Implementation notes

The help system for ODIP Space is not a static manual behind a button. It is a genuinely context-aware system that knows where the user is, who they are, and what version context they are in — and delivers help accurate to that exact combination. The architecture and UX decisions below are final.

UX principle — tooltips on labels, never on values

There are no ? icons next to fields. The application must not be cluttered with help markers. Instead, any labelled UI element carries a tooltip on hover. This covers: page titles, section titles, field labels, query area titles, query field labels, and column headers. The label is always the hover target — never the value in a field or a cell in a result set. A result set cell is a value; the column header already contextualises it.

The hover delay must be long enough (~1 second) that users navigating or working quickly are never disturbed by accidental tooltip triggers.

The tooltip contains a short, role-invariant Purpose-level text — one or two sentences. It also carries a "More..." link that opens the full contextual help. The "ODIP Space" title in the Home page hero band is itself a labelled element — its "More..." is the entry point to the help site for new users discovering the application.

Two-level help pattern

LevelTriggerContentRole variation
Level 1 — TooltipHover on any label (~1s delay)Short purpose text (1–2 sentences) + "More..." linkNone — same for all users
Level 2 — Full helpClick "More..." in tooltip, or ? Help button in headerFull explanation: rules, recommendations, role-specific guidance, examplesYes — role + version context modifiers applied

External singleton — implementation specification

Full help opens in a separate browser tab/window, reused as a singleton. The named window target gives the singleton for free — if the help window is already open, it navigates to the new anchor rather than opening a second window. The user manages positioning; ODIP Space does not attempt to control the help window's position or size (browser security restrictions prevent this reliably). This approach has zero impact on ODIP Space layout and suits the constrained 1280px viewport.

// Module-level variable — holds the help window reference
let helpWindow = null;

// Called on every help trigger (tooltip "More..." or ? Help button)
function openHelp(anchor) {
  const url = HELP_SITE_BASE_URL + '#' + anchor;
  if (!helpWindow || helpWindow.closed) {
    helpWindow = window.open(url, 'odip-help');
  } else {
    helpWindow.location.href = url;
    helpWindow.focus();
  }
}

// ? Help button in header → page-level help
openHelp(currentPage);

// Tooltip "More..." link → field-level help
openHelp(currentPage + '-' + fieldId);

9.2 — O*s tab

9.2.1 — O*s area — query form

The O*s area presents a query form followed by a result set. The query form is the mandatory entry point — no result set is shown until the user submits a query. Active users are power users. The query form is designed for deliberate, intentional use. No over-protection, no hand-holding. Tooltips on all field labels per the contextual help architecture.

Empty = no filter. Empty query fields mean no filter applied. There is no explicit "All" option anywhere. All fields combine with AND logic. Each active field narrows the working set. Dismiss: Escape or click outside closes any open picker without selecting.

9.2.6 — O* detail pages — ON, OR, OC

Navigation pattern + field layout

Navigation — universal stack model

O* detail pages follow the universal navigation model defined in the Navigation model section. Clicking any O* link — from a result set, a plan view, a wave page, the evolutions panel, or any other context — pushes the O* detail page on top of the current stack. Closing it (× or ← Back) returns to exactly the previous state.

When reached from a result set, the user can walk through the result set in gallery mode using large naked chevron arrows fixed to the left and right viewport edges, vertically centred, always visible regardless of scroll. Each arrow shows the position number of the adjacent item (e.g. ‹ 2 on the left, 4 › on the right). The sub-navigation bar shows the current position ("3 of 12") and a ← Back to result set link. Walking replaces the detail page at the top of the stack — it does not push a new level. Walking order follows the result set sort order (ascending by wave, then ON→OR→OC within same wave).

When not reached from a result set (e.g. via a link from another detail page), there are no walking arrows and no position counter — only ← Back.

Result set — flat list and tree view

The result set is a flat list only — no side panel. Clicking a row opens the detail view. Two display modes are available: Collection (flat list, all columns) and Tree (hierarchical grouping by domain → ON → ORs). Both modes support the same walking pattern. Tree mode is particularly useful for domain writers who think in terms of their domain's ONs and implementing ORs.

9.2.6a — OR Dependencies tab

Capability catalogue · requires/produces · AI-assisted discovery · governed workflow · NM internal only
Design status: Fully specified and mockuped in v1.31. Remaining open: capability evolution/versioning model (produces vs. evolves); picker for manually adding a dependency; batch hint generation surface in Management (§0.4 backlog).

Why OR dependency maintenance is hard

Inter-OR dependencies are essential input to the prioritisation process — an OC cannot be assigned to a wave before the OCs implementing its dependencies. Yet building a correct and complete dependency graph is one of the hardest recurring activities in ODIP Space. Several compounding factors explain the difficulty:

  • Cross-domain knowledge gaps. Domain writers know their own domain's ORs well but rarely have sufficient knowledge of other domains' ORs to recognise dependencies across domain boundaries.
  • Volume. With potentially thousands of ORs in the dataset, no individual can hold the full corpus in mind when authoring a new OR.
  • Operational-level reasoning is insufficient. Reading two OR statements and deciding whether one genuinely cannot be implemented without the other requires reasoning about how things get built, not just what operations need. A dependency is ultimately an implementation sequencing constraint — a purely operational reading of OR text will not reliably surface it.
  • Granularity variance. OR granularity varies significantly across DrGs and individuals. A coarse-grained OR in one domain may span the scope of five fine-grained ORs in another, making dependency relationships harder to articulate cleanly and sometimes raising the question of whether the right level of decomposition has been chosen.
  • Thematic relatedness vs. genuine dependency. Two ORs can be closely related — addressing the same domain, referencing the same data, serving the same stakeholders — without one being a prerequisite for the other. This distinction is subtle and hard to resolve from OR text alone, even for domain experts.

AI assistance has been explored for dependency discovery. Providing the full OR corpus to an AI yields poor results for a specific reason: the AI can detect thematic relatedness reliably, but cannot reliably distinguish it from genuine implementation dependency. The missing ingredient is not subject-matter knowledge — AI has sufficient ATM operational context — but rather a structured basis for implementation reasoning.

The solution concept: capability catalogue + requires/produces

The approach is based on making implementation reasoning explicit and structured, rather than leaving it implicit in OR text. Three elements work together:

Implementation hint. A short, informal, NM-internal note on each OR sketching how it is likely to be implemented and what it will need — "this OR will require a persistent flight object store and a real-time notification channel to AOs." Not normative, not a technical spec. Its sole purpose is to give the AI and human reviewers a richer basis for reasoning about sequencing. AI drafts this hint from the OR statement and rationale; domain writers and integrators review and correct it.

Capability catalogue. A managed set of named capabilities — things that can be produced by implementing an OR and required as a precondition by another. Examples: "live flight object store", "real-time AO notification channel", "AIRAC baseline diff API". Each entry has a short name and a description sufficient to distinguish it from near-duplicates. The catalogue is a setup element, maintained collaboratively by domain writers and integrators. It is NM-internal only — never shown to externals.

Requires / Produces associations. Each OR carries two optional lists referencing catalogue entries: Requires (capabilities that must exist before this OR can be implemented) and Produces (capabilities that this OR's implementation makes available to others). AI proposes these associations from the implementation hint and OR text; domain writers and integrators confirm them.

With these associations in place, candidate dependencies become mechanically derivable: if OR-A requires capability C and OR-B produces capability C, then OR-B is a candidate prerequisite of OR-A. No AI reasoning is needed at this step — it is a graph query. The AI's role is confined to the two well-defined tasks it handles well: drafting implementation hints, and proposing requires/produces tags.

The dependency maintenance workflow

The workflow runs in two phases: per-OR preparation (asynchronous, ongoing) and cross-OR review (periodic, governed). Known traps

Trap
Description and mitigation
Capability granularity drift
If capabilities are too coarse ("data exchange"), the graph becomes saturated — everything depends on everything and the result is useless. If too fine ("field X in message type Y"), the catalogue explodes and near-duplicates multiply. Granularity drifts naturally as different domain writers propose entries at different levels of abstraction. Mitigation: integrators own granularity discipline. A working principle: a capability should be something a wave could deliver independently — not a field or message type, not an entire system. The Quality page surfaces catalogue health indicators (duplicate candidates, entries with no OR associations, entries associated to only one OR).
Produces vs. evolves
Some ORs do not create a capability from scratch — they evolve an existing one. OR-B may produce "live flight object store v1" and OR-C produce "live flight object store v2". An OR-A requiring "live flight object store" will be flagged as a candidate dependency of both OR-B and OR-C, but the correct dependency is only on OR-B (v1 is sufficient). Naively, the graph produces false dependencies. Mitigation: capability entries should represent stable, version-independent abstractions. "Evolves" relationships between capability versions should be expressible, so that the graph query can resolve the minimum sufficient version. This requires further design — noted as open.
Asymmetric knowledge
The domain writer on the requires side may not know that another OR produces the needed capability — they may not even know the capability exists in the catalogue yet. Conversely, the producing side may not realise their OR is a dependency blocker for others. Discovery is inherently asymmetric. Mitigation: the Dependencies Maintenance surface (to be designed) must actively surface Pending pairs to both sides — not rely on either party to go looking. Notification to both sides when a new Pending pair is generated.
Confirmation deadlock
Step 5 requires both sides to confirm. If the producing-side domain writer disagrees ("my OR does not produce that capability") or simply does not respond, the pair can block indefinitely. Mitigation: integrators can override and confirm or reject unilaterally after a defined period, with reason recorded. Pending pairs older than a threshold are surfaced in the Quality page as stale. Disagreements (one side Confirmed, other side Rejected) are flagged explicitly and escalate to integrator resolution.
Granularity mismatch between ORs
A coarse-grained OR in one domain may span the scope of several fine-grained ORs in another. A dependency declared between them may be imprecise — the coarse OR only partially requires the producing OR, or the dependency should be at ON level rather than OR level. Mitigation: when confirming a pair, reviewers can add a clarifying note. The process does not prevent declaration of imprecise dependencies but makes the imprecision visible. Structural resolution (splitting an OR, or elevating the dependency to ON level) remains a manual decision by the domain writer and integrator.

AI trigger — on demand only
Hint generation is triggered by an integrator clicking "Generate hint + propose tags". Not automatic on OR save. One call produces the hint and proposes requires/produces tags simultaneously, using the full catalogue as context.
Automatic generation on every save would accumulate cost proportional to the full corpus, for hints that may sit unreviewed for months. On-demand keeps cost bounded and deliberate. Combining hint and tag proposal in one call is more efficient and ensures the tags are grounded in the same implementation sketch.
Hint generation — integrators only
The "Generate hint + propose tags" action is available to integrators only, not domain writers.
Integrators are responsible for catalogue granularity discipline and cross-domain consistency. Concentrating the generation trigger with integrators ensures the catalogue is mature enough before hints are proposed against it, and avoids unchecked AI calls by many concurrent users.
Dependency provenance
One entry per OR pair. Each entry shows its provenance: "manually declared", "via [capability name(s)]", or both. Multiple catalogue paths to the same pair are merged into one entry with multiple capability tags.
Reviewers must be able to distinguish dependencies introduced by a human decision from those derived mechanically by the catalogue process — and to know exactly which capability(ies) produced each derivation. Merging into one entry per pair avoids duplicates and keeps the list readable.
Capability catalogue location
Capability catalogue is a Reference data entry in the Management nav. Integrators manage it there: add, edit, merge entries; monitor health indicators (orphans, duplicate candidates).
The catalogue is a shared vocabulary, not per-OR data — it belongs with other reference data managed centrally by integrators. Management is the right home. No separate Setup label — that term is retired.

9.2.6b — Time Traveller — design

The Time Traveller is a dedicated page pushed onto the stack when the user triggers the History action on an O* or wave detail page. It is a version gallery — the same gallery-mode mechanic used for walking through a result set, applied to the version history of a single object. The CFT marks diff display as critical: "O* evolutions are the primary interest of ODIP users." The Time Traveller is the primary surface for that.

ONs, ORs, OCs, and waves are all versioned symmetrically using the same append-only model. Each save creates a new version; all previous versions are permanently stored. The Time Traveller works identically for all four types.

Entry points and landing position

Entry pointLanding position
History action on O* or wave detail pageLatest version (vn of n). The user knows where they are and walks backward from there.
Versioned deep link (email, conversation ↗, any external source)The specific version encoded in the link. Walking available in both directions from there.

Two chevron axes

ChevronsMovementNotes
< / >One O* version at a timeFine-grained. Steps through individual saves in the append-only history. Always available.
<< / >>One ODIP edition at a timeCoarse-grained. Jumps to the O* version captured in the adjacent published edition or baseline. Answers "what did this look like in Edition 2025 vs Edition 2026?" without needing to know version numbers. Available when at least one published edition exists.

Both axes replace the top page — they do not push a new level. The two axes are independent and can be used in any order. Walking forward beyond the latest saved version is not possible; walking backward beyond v1 is not possible. The double chevrons are greyed when no further ODIP edition exists in that direction.

Travelling to the future. When the active ODIP context is a published edition, versions saved in the dataset after that edition was published are "future" versions relative to the edition. The Time Traveller spans the full version history regardless of the current ODIP context — walking forward with > may bring the user to versions that do not exist in the currently selected edition. This is intentional and useful: it lets a user see where an O* is heading without switching context.

Version counter and jump picker

The sub-nav bar shows a version counter, e.g. v4 of 6. The counter is clickable and opens a compact dropdown listing all versions with date, author, and maturity at save — the jump picker. Clicking any row in the dropdown jumps directly to that version without stepping through intermediate ones. The current position is highlighted in the dropdown. This is the fast-access mechanism for objects with many versions.

Stack page name

The Stack indicator label for the Time Traveller is the O* or wave identifier, suffixed with " — History". Example: AOP/NOP Integration — History. This distinguishes it from the underlying O* detail page which shows just the title.

Time Traveller in the navigation stack

The Time Traveller occupies exactly one level in the page stack:

  1. User is on an O* or wave detail page at stack depth N.
  2. History pressed → Time Traveller pushed onto stack (depth N+1), opening at the latest version.
  3. Chevron walking (< / > or << / >>) → top page replaced in place. Stack depth stays at N+1 throughout all walking.
  4. ← Back from any version stop → Time Traveller removed, stack returns to depth N, always landing on the original O* or wave detail page regardless of which version was being viewed.

Walking does not push new levels — the Time Traveller replaces its own top position, the same mechanic used when walking through a result set.

Diff mechanism — decided v1.30. Purely display-side; no data model changes required. Each rich text field is compared paragraph by paragraph. Each paragraph is classified as unchanged, removed, or added:

  • Unchanged — shown muted for context. Never hidden: surrounding paragraphs give meaning to changed ones.
  • Removed — red tinted background, struck through, labelled “removed”.
  • Added — green tinted background, labelled “added”.

A rewritten paragraph appears as one removal + one addition. List items (<li>) are paragraph-level elements — same tombstone logic per item. Structured fields (chips, links, cost tables, OC assignment lists) are diffed at item level.

Image blocks in rich text. Each uploaded image receives an opaque internal ID at upload time. Comparison uses this ID: same ID = unchanged; different ID = replacement, regardless of visual similarity. No pixel comparison. When an image is replaced, three presentation options are available — decision deferred to development time based on what the rich text renderer makes easiest:

  • Option A — full tombstone. Old image at natural width in red frame (struck through), new image below in green frame. Consistent with the paragraph model, no special rendering case. Acceptable height cost given images are rare in these fields.
  • Option B — thumbnail tombstone. Same logic, images constrained to ~160px wide with a “view full size” link. Compact; introduces a special rendering path for images vs. text.
  • Option C — filename placeholder. No image rendered — only filename and internal ID in a tombstone row. Lightest; gives no visual information about the nature of the change.

Deep linking into the Time Traveller. A versioned deep link encodes object type + object identifier + version number. When followed, the Time Traveller opens at the specified version. Optionally, an ODIP edition identifier may be encoded instead of a version number — the application resolves the mapping to the corresponding O* version. See the Deep linking section.

O* detail — top bar actions (create, delete, decommission)

The top navigation bar of the O* detail page carries five actions, always in this order: Edit · History · separator · Decommission · Delete · Delete version. Edit is the primary positive action. History pushes the Time Traveller onto the stack. The three consequential actions follow the separator. All are always visible; Delete and Delete version are greyed when the O* has appeared in any past published edition.

Creating a new O*

A + New O* dropdown button appears both above and below the result set in the Elaboration O*s area (dataset only, active users only). When no query has been submitted and no result set exists, it appears once — in the query form footer. The dropdown offers three choices: ON, OR, OC. Selecting one opens an empty O* detail page in edit mode, pushed on top of the Elaboration page. The type is pre-set and the code pre-assigned. There is no walking context (no position counter, no chevron arrows). The amber "Editing" bar shows "New ON" (or OR / OC) alongside the Editing badge. Cancel returns to Elaboration without creating anything. Save creates the O* and returns to Elaboration.

Backlog — avalanche decommission. Cascading decommission of dependent O*s (e.g. decommissioning an ON automatically proposes decommissioning its ORs and their OCs) to be considered in a future iteration.

9.3 — Plan tab

The Plan tab is the ultimate output of ODIP Space — everything else (authoring, quality work, prioritisation) exists in service of producing a plan that users can compose, navigate, and consume. It has three sub-tabs: Waves, Bandwidth, and Calendar.

9.3.1 — Plan tab structure

The Plan tab serves two distinct purposes that are accessed differently.

The plan view area is the primary surface — a visual and textual rendering of the full ODIP plan. It is always visible when the Plan tab is open, above the sub-tabs. It supports two modes: pre-cooked views (curated, immediately available) and the user-composed view (driven by the active O*s result set). Both modes are described in §9.3.2.

The three sub-tabs below are wave management tools — they do not replace the plan view, they complement it.

9.3.2 — Plan design principles

The Plan is the ultimate output of ODIP Space. All authoring, quality work, and prioritisation exists in service of producing a plan that users can compose, navigate, and consume. The plan view area at the top of the Plan tab is where this happens.

Two renderings, same data

The plan view is always available in two renderings, user-selectable: textual (structured table — waves as rows, OCs listed within each wave, filterable) and visual (Gantt-like timeline — same data rendered spatially with zoom). Both renderings show the same underlying plan data and support the same zoom levels and navigation.

Four zoom levels

Decade view — top-level ONs plotted against their tentative implementation time spans across the full planning horizon. Stratospheric read for executives and strategists. Period view — one to three year window, waves visible as discrete deployment units, OCs listed within each wave. Filterable by domain or stakeholder category. Year view — single year, all waves and their OC assignments in detail. Sub-year view — zoom into a specific wave: OCs, milestones, iCB report summary. All four zoom levels are available in both textual and visual renderings.

Pre-cooked views

For users who do not want to query — executives, occasional visitors — the plan view area provides a library of curated pre-cooked views (e.g. full decade view, current year, domain-specific). These are saved queries with a plan rendering, prepared by integrators or iCDM. The user opens the Plan tab and immediately sees something meaningful without touching the query form.

User-composed plan

When the user has an active O*s result set, the Plan tab offers a user-composed view alongside the pre-cooked ones. The result set is the selection context: the plan shows the full temporal landscape but emphasises the selected O*s, with surrounding context dimmed. The six document export formats (Word / PDF / Static site / CSV / XLSX / JSON) are outputs of that same result set — the plan view is the live interactive rendering of it. When no result set is active, only pre-cooked views are available in the plan view area.

Full navigability

The plan view is navigable at every zoom level. Clicking an ON band opens the ON detail page. Clicking a wave opens the Wave detail page. Clicking an OC opens the OC detail page. The plan is a navigation surface, not a static image. Zoom, filter, and navigation are available at all times.

9.3.7 — Prioritisation process

Overview

The prioritisation process produces the ODIP plan — all Operational Changes assigned to waves, with bandwidth respected and dependencies satisfied. It combines two steps: OC preparation (asynchronous, by domain writers and integrators) and iterative prioritisation (in iCDM sessions, by integrators and iCDM).

The Waves sub-tab is the primary tool for Step 2. The Bandwidth sub-tab supports it. Domain writers are present at iCDM sessions and contribute to the discussion but do not operate the tool — wave assignment is restricted to integrators and iCDM.

9.3.7.1 — Step 1 — OC preparation

Done asynchronously by domain writers (within their own domain) and integrators (any domain), outside of iCDM sessions. This step feeds the inputs the prioritisation algorithm and iCDM will use.

ActivityWhoWhere
Build OCs — add ORs, enter per-OR cost in MWDomain writers, integratorsOC detail page — Implemented ORs table
Set tentative implementation time on ONsDomain writers, integratorsON detail page
Declare OR dependenciesDomain writers, integratorsOR detail page — Dependencies field
Set wave bandwidth budgetsIntegrators, iCDM onlyBandwidth sub-tab

The OC total cost is the read-only sum of its per-OR costs. It is visible to active users only (domain writers, integrators, iCDM) — never shown to passive users or externals. At the end of Step 1, the backlog drawer contains all unassigned OCs, each carrying a cost, a maturity, and a domain — ready for prioritisation.

9.3.7.2 — Step 2 — Iterative prioritisation

Typically takes place in iCDM sessions, with integrators and iCDM operating the Waves sub-tab. Domain writers attend and contribute to the discussion. The process is explicitly iterative — there is no single "done" moment. The iCDM converges toward a balanced plan through repeated cycles of assignment, re-proposal, and adjustment.

Three mechanisms are available in combination:

MechanismDescription
Manual assignmentDrag-and-drop OCs between the backlog drawer and wave rows, or between wave rows. Immediate effect, Undo available. Integrators and iCDM only.
Re-proposalThe application proposes a set of OC assignments based on tentative implementation times, dependencies, and cost vs bandwidth. Presented as a negotiation dialog — the iCDM accepts or rejects each suggestion individually before anything is applied.
Pin / unpinA pinned OC assignment records a deliberate iCDM decision. The re-proposal algorithm respects pins as constraints but can challenge them if the situation has changed, clearly signalling the challenge in the dialog.

Cost editing and bandwidth adjustment are available at any time during Step 2 without leaving the Waves sub-tab — cost via a popover on the OC chip, bandwidth by switching to the Bandwidth sub-tab (integrators and iCDM only).

9.3.7.6 — Re-proposal dialog

  • Tentative implementation times set on implementing ONs
  • Dependency constraints between OCs (fully resolved transitively)
  • Cost vs bandwidth — fitting OCs within available wave budgets
  • Existing pinned assignments — respected as constraints, but challenged if the situation has changed

The dialog presents the suggestions as a list. The iCDM reviews each row and accepts or rejects it. Nothing is applied to the plan until the final Apply action.

11 — Conversations

Five subject types · access model · version anchoring · moderation · tab design

Overview

A conversation is a threaded exchange anchored to a specific subject. Five subject types exist: the application as a whole, a version as a whole (dataset or edition), an O*, a wave, and a conversation itself. The last type supports targeted collaboration — pointing a colleague to a specific thread is simply starting or continuing a conversation anchored to it.

Conversations accumulate over time. As their subject evolves — new O* versions, new editions, new wave compositions — the application compiles contributions into a single continuous thread, inserting version separators to mark the version context in which each contribution was made. A conversation is never automatically closed by a version transition. Closure is always a manual act by the originator or an integrator.

The application as a whole is a feedback channel: bug reports, feature requests, general impressions. Any user may initiate; NM staff respond. No gating of any kind applies to this subject type.

Access and visibility

Two rules govern what users see.

Initiation visibility: a conversation initiated by an NM user (active user or NM reviewer) is internal — never visible outside NM, regardless of subject type. A conversation initiated by an external user is visible to all users who have access to that subject.

Response visibility: NM responses within an externally-initiated conversation are visible to all. This is how NM engages publicly with external feedback — not by initiating, but by responding. NM identity (name) is never exposed to external users in any context, including tombstones of deleted responses.

Review window

NM reviewers are subject to an additional constraint on dataset O*s and waves: write access is gated by a review window opened and closed by integrators. Outside an open window, NM reviewers can read dataset conversations but not contribute. The review window exists to protect authors from being burdened with review comments in the middle of active authoring work.

The review window does not apply to dataset-as-a-whole conversations, conversation-on-conversation, or any published edition context. It also does not apply to DRAFT O*s — NM reviewers cannot see DRAFT O*s at all, regardless of window state. Within the dataset, DRAFT O*s are visible to active users only.

Moderation

Integrators act as moderators. Four moderation actions are available:

  • Suspend — prevents further contributions, marks the conversation as discontinued, excludes it from pending indicators. Reversible.
  • Reactivate — reverses a suspension. Available to integrators and the conversation originator.
  • Delete a response — soft-deletes a single contribution. The thread remains intact; the deleted item is replaced by a tombstone. The tombstone text is role-dependent: NM users see "Response deleted by [name] on [date]"; external users see "Response deleted on [date]" — NM identity is never exposed outside NM. Integrators, the author of the deleted response, and members of the author's organisation (when both the visitor's and the author's organisation attributes are set and match) can reveal the original content inline via a Show action, clearly marked as deleted. Soft-deleted responses are restorable by integrators.
  • Delete a conversation — soft-deletes the entire thread. Restorable by integrators. Hidden from normal operational views but retained for accountability.

Suspend and reactivate are available to integrators and the conversation originator. Delete a response is available to integrators and the author of that response. Delete a conversation is available to integrators and the originator. All moderation actions are logged.

Each conversation displays a persistent visibility indicator — stating clearly who can see it — positioned at the top of the thread, always visible before reading any content.

The Conversations tab

The Conversations tab sits in the nav bar. It is version-neutral and always accessible regardless of whether a version is selected. Its purpose is to find, monitor, and manage conversation threads across the application. Conversations are not primarily initiated from this tab — they are initiated in context, close to the content being discussed: on the O* page, the wave page, the version description area, or the application feedback surface. The tab is where users go to query and retrieve threads across the whole application.

The tab follows a master/detail pattern, self-contained without stack disruption. Level 1 (the list) and Level 2 (the thread) are always both visible. Arriving from a contextual surface — e.g. opening Conversations from an O* page — lands directly on Level 2 with the relevant conversation selected and Level 1 pre-filtered to that subject. A back action resets the filter and returns focus to the full list.

Level 1 — conversation list. Default sort: pending first (external conversations awaiting NM response, plus internal conversations unacknowledged by the current user), then recency. Internal and external conversations are interleaved, each row badged accordingly. Filter chips: All · Pending · Internal · External · subject type filters (O* · Wave · Version · App). Free text search. When a context filter is active (arriving from a subject), a clear indicator shows the active filter with a one-click reset.

Level 2 — conversation thread. Subject and version anchor displayed in the nav area, with a ↗ link to open the subject object. The ↗ link opens the Time Traveller for that object, landing at the specific version anchored to this conversation — regardless of which ODIP version is currently selected. If that version is the latest, the Time Traveller opens at the latest version; if it is a past or future version, the Time Traveller opens at the anchored version with walking available in both directions. Version separators are embedded in the thread to mark version context at each contribution. Visibility indicator persistent at top of thread. Acknowledge action available; last acknowledgement shown. Compose area for responses.

The Stack indicator label for the Conversations tab is "Conversations".

11b — Deep linking

Versioned links · URL format · access rights · prior closure reversed

Prior closure reversed

An earlier design session concluded that URL-based deep linking was not the right solution for internal users, on the grounds that (1) linking to the current state of an O* produces a drifting target — the content at that URL changes as the O* is edited — and (2) the sharing need was better served by private conversations with version anchoring. Both grounds have since been superseded: (1) the Time Traveller introduces versioned links that are permanent and immutable — v3 of ON-ARPT-0042 will always resolve to the same content; (2) private conversations have been removed. Deep linking is fully rehabilitated for versioned links.

What a versioned deep link is

A versioned deep link encodes a specific object at a specific version. It is permanent and unambiguous — the linked content does not change over time. Following a versioned deep link opens the Time Traveller for that object, landing at the encoded version. Walking is available in both directions from that landing position.

Links to the current (unversioned) state of an O* are not supported as shareable deep links, precisely because the target drifts. The Time Traveller always operates on a specific version — including the latest, which is itself a specific version at the moment the link is generated.

URL format

To be defined. The link encodes: object type (on / or / oc / wave) + object identifier + version number. Example structure: /tt/on/ON-ARPT-0042/v3. Optionally, an ODIP edition identifier may be encoded instead of a version number — the application resolves which O* version was captured in that edition.

Exact URL design, including path structure and query parameter conventions, to be finalised during implementation.

How the sender generates a link

The Time Traveller tt-bar includes a ⊕ Copy link to vN button on the right side, always visible at every version stop. Clicking it copies the versioned URL for the currently displayed version to the clipboard. The version number in the label confirms exactly which version is being shared. The button is present at all versions — including the latest — so users do not need to navigate to a specific version before copying; they can copy from whatever stop they are at.

Access rights mismatch

Two distinct cases exist, each with its own message. In both cases the message explains why access is restricted — not merely that it is. No information about the object content is revealed.

CaseCauseMessage shown
Post-edition version The linked version was saved after the edition the user is currently browsing. It exists in the dataset but not in this edition. “This version of [object title] was saved after Edition [year] was published and is not part of it. Switch to the dataset or a later edition to view it.”
Domain access restriction The object belongs to a domain the user has no read rights on (applies to external users and domain writers from other domains). “You do not have read access to the [domain name] domain. This object exists but cannot be displayed with your current rights.”

Scope

Versioned deep links apply to O*s (ONs, ORs, OCs) and waves — the four types covered by the Time Traveller. Conversations are addressable objects and may also be linked directly (landing on the conversation thread in the Conversations tab). ODIP editions as a whole are not deep link targets.

12 — Management

Version-independent · left nav · integrators only · Editions · Admin

Management is the administration area of ODIP Space. It is accessible only to integrators — the tab is visible but greyed for all other user categories. Management is version-independent: it is always accessible regardless of whether a version is selected, and its content does not change based on the selected version. Dataset entities (domains, stakeholder categories, waves, etc.) are managed in the Setup tab of Elaboration — not here. Management covers only what cannot belong to any specific version: edition lifecycle governance and application administration.

Page structure

Management uses a two-pane layout with a persistent left nav and a content area. The left nav lists all Management sections as collapsible groups; clicking a section opens it in the content area on the right. There is no stack navigation within Management — the left nav replaces it. Two groups:

Editions — edition lifecycle governance. Sub-items: Edition list · Baseline creation · Publish / unpublish · Patching. These actions operate on the edition record system, not on any specific version's content.

Admin — application administration. Sub-items: Monitoring · Users · Log · Backup. Monitoring covers distribution list addresses and alerting configuration. Users covers user management and access rights. Log provides the audit trail and backup file download. Backup covers backup and restore operations.

Stack page name

The Management area is named Management in the stack. Sub-items follow the pattern Management — [Section] (e.g. Management — Baseline creation, Management — Users).
